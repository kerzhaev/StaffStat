## ✅ Задача 1 — Расширение Health Check: проверка пустых критичных полей

**Где:** `mod_Maintenance_Logic.bas` → `RunDataHealthCheck`

**Что сейчас:** Health Check проверяет только дубликаты PersonUID, дубликаты FullName+BirthDate, сирот в tbl_History_Log и будущие даты.

**Что добавить:** Проверку на пустые обязательные поля в `tbl_Personnel_Master` — например, `FullName`, `BirthDate_Text`, `PersonUID`. Записи с пустыми критичными полями пишутся в `tbl_Validation_Log` с типом `EmptyRequiredField`.

**Результат:** Более полный контроль качества данных после каждого импорта.

## ✅ Задача 2 — Экспорт отчёта об изменениях с авто-сохранением файла

**Где:** `mod_Reports_Logic.bas` → `GenerateAuditReport` и `GenerateCurrentStaffReport`

**Что сейчас:** Оба отчёта открывают Excel-книгу, но **не сохраняют** её на диск автоматически — пользователь должен сохранять вручную.

**Что добавить:** Автоматическое сохранение файла в папку `Exports\` с именем вида `AuditReport_20260208.xlsx` или `Snapshot_20260208.xlsx` сразу после формирования, с показом пути в сообщении.

**Результат:** Отчёты сохраняются без участия пользователя, папка `Exports\` уже существует в проекте.

## ✅ Задача 3 — Поддержка нескольких профилей маппинга в UI

**Где:** `mod_Import_Logic.bas`, `tbl_Import_Mapping`, `tbl_Import_Profiles`, `uf_Settings`

**Что сейчас:** Импорт всегда использует `ProfileID = 1` (жёстко прописан). Таблица `tbl_Import_Profiles` уже существует и предназначена для хранения нескольких профилей.

**Что добавить:** В форме настроек (`uf_Settings`) — выпадающий список для выбора активного профиля. Выбранный `ProfileID` сохраняется в `tbl_Settings` (ключ `ActiveImportProfile`). `ImportExcelData` читает этот ключ и передаёт ProfileID в `RunDynamicImport` вместо константы `1`.

**Результат:** Возможность работать с разными форматами Excel-выгрузок без изменения кода.

## ✅ Задача 4 — Статистика изменений на Dashboard по периодам

**Где:** `mod_Maintenance_Logic.bas` → `GetDashboardStats`, `uf_Dashboard`

**Что сейчас:** Dashboard показывает только `TotalCount`, `ActiveCount`, `ErrorCount`. Поля `txtStartDate` / `txtEndDate` уже есть на форме, но используются только для отчёта об изменениях.

**Что добавить:** Новый счётчик `ChangesInPeriod` — количество записей в `tbl_History_Log` за выбранный период (между `txtStartDate` и `txtEndDate`). Отображать рядом с существующими метками на Dashboard. Обновляется при смене дат или нажатии кнопки.

**Результат:** Dashboard становится настоящей панелью мониторинга с динамической статистикой.

## ✅ Задача 5 — Модуль автоматического резервного копирования по расписанию

**Где:** `mod_Maintenance_Logic.bas` → `CreateDatabaseBackup`, `tbl_Settings`

**Что сейчас:** Бэкап создаётся только вручную. В `tbl_Settings` уже есть инфраструктура для хранения настроек (`AutoCheckEnabled` используется как пример). Папка `Backups\` уже создаётся автоматически.

**Что добавить:** Настройки `BackupEnabled` (True/False) и `BackupIntervalDays` (число дней) в `tbl_Settings`. При каждом запуске приложения (`mod_App_Init`) — проверять дату последнего бэкапа (`LastBackupDate` в `tbl_Settings`) и, если прошло больше `BackupIntervalDays` дней, автоматически вызывать `CreateDatabaseBackup` в тихом режиме.

**Результат:** Защита данных без участия пользователя, настраиваемая через уже существующий механизм Settings.
